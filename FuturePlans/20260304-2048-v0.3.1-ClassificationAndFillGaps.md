## Context

v0.3.0 introduced border-segment classification (Step D0 + E3 constraint enforcement) and the `mergeSmallComponents` post-processing utility. Results are significantly improved over the original global-nearest-segment approach, but residual issues remain — particularly for multi-crossing open surfaces like the Kirra Convoluted block.

### Key Problems

1. **Classification spill at crossings**: A few triangles adjacent to intersection segments end up on the wrong side. The border-segment fix (D0) catches most, but some sub-triangles sharing only a vertex (not an edge) with a segment get classified by the fallback D1-D4 path, which can pick the wrong nearest segment for multi-crossing geometry.

2. **B components over-merged**: The convoluted block crosses the terrain twice, creating 2 separate B-inside and 2 separate B-outside regions. The `mergeSmallComponents` threshold (50) absorbs the smaller B regions into the larger ones. The threshold needs to be smarter — potentially relative to the total group size, or using spatial separation to decide whether to merge.

3. **Fill gaps not selectable**: `fillOpenEdgeLoops` fills ALL detected open edge loops. For user-picked custom surfaces, some open edges are intentional (terrain boundaries). The library needs to let the caller choose which loops to fill.

## Work Items

### WI-1: Improve Sub-Triangle Classification for Multi-Crossing Meshes
**Priority:** High
**Files:** `src/boolean/classifyTriangles.js`

The core issue: when a mesh crosses another mesh at multiple locations, the `halfSpaceTest` fallback (D1-D4) picks the globally nearest segment. For a sub-triangle between two crossings, the "nearest" segment might be from the wrong crossing, giving wrong classification.

**Approach A — Segment-Constrained Flood Fill:**
After the border-segment classification (D0) assigns confident classifications to triangles touching segment edges, flood-fill outward from those confident triangles through edge adjacency, but STOP at any edge that is an intersection segment. This ensures classification only propagates within a single "cell" between crossings, never jumping across a segment boundary.

**Approach B — Per-Crossing-Region Classification:**
1. Use the intersection segment chains to partition the parent mesh into regions (connected components bounded by segment chains)
2. Within each region, classify independently using only segments from that region's boundary
3. This ensures each crossing is handled locally

**Recommended:** Approach A is simpler and builds on the existing infrastructure. Approach B is more robust but requires significant refactoring.

### WI-2: Smarter Merge Threshold
**Priority:** Medium
**Files:** `src/boolean/booleanOp.js`

Replace the fixed triangle-count threshold with a smarter heuristic:
- **Option 1: Relative threshold** — merge components smaller than X% of the largest same-group component (e.g., 5%)
- **Option 2: Spatial gap detection** — only merge if the small component's bounding box overlaps or is within one average-edge-length of the target component
- **Option 3: User-configurable** — expose threshold as a parameter and let the caller decide. This is already partially done (threshold param) but needs better defaults.

For B-component over-merging: the issue is that with a double-crossing, B-inside has two legitimate regions of ~170 and ~185 tris. At threshold=50, neither gets merged (good). But B also has tiny fragments (5, 5, 2 tris) that merge into the larger. The real fix is WI-1 — if classification is perfect, B will have exactly 2 components per group with no fragments.

### WI-3: Selectable Fill Gaps
**Priority:** High
**Files:** `src/repair/fillOpenEdgeLoops.js`, `src/index.js`, `src/index.d.ts`

Current `fillOpenEdgeLoops(soup)` finds and fills all open-edge loops. Needs to become:

```javascript
// Step 1: Detect loops (non-destructive)
var loops = detectOpenEdgeLoops(soup);
// loops = [{ edges: [...], centroid: {x,y,z}, perimeter: number, edgeCount: number }, ...]

// Step 2: User selects which loops to fill (e.g., via UI)
var selectedLoops = [loops[0], loops[2]]; // user picks

// Step 3: Fill only selected loops
var filled = fillSelectedLoops(soup, selectedLoops);
```

**Demo integration:**
- In the Custom view, when "Fill Gaps (Fan)" is checked, show a list of detected loops with eye toggle icons (like components)
- Each loop gets a distinct colour wireframe preview in the scene
- User toggles which loops to fill
